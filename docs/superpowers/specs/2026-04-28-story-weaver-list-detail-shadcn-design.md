# Story Weaver 列表详情式 shadcn 统一化设计

## 目标

将当前 Story Weaver 渲染层重构为基于 `shadcn/ui` 官方风格的统一桌面工作台，并将核心信息架构调整为更标准的“左侧导航 + 作品列表/详情”模式。

本次设计的目标不是局部换皮，而是完成以下三件事：

- 主题风格直接回归 `shadcn` 现有主题体系，不再继续扩展一套自定义视觉语言
- 基础交互组件尽量使用 `shadcn/ui` 官方生成组件，而不是继续维护手写仿制版本
- 页面结构从“纵向堆叠多个模块”重组为“导航驱动的工作台布局”，其中作品页采用列表详情结构

## 已确认决策

### 1. 视觉与组件方向

- 主题直接 follow 当前仓库已配置的 `shadcn` 基线
- 使用 `components.json` 中既有配置：
  - `style: default`
  - `baseColor: slate`
  - `cssVariables: true`
- 基础组件优先通过 `shadcn` 官方安装和生成
- 不额外发明新的主题系统或品牌化视觉语言

### 2. 页面改造范围

- 本次目标是统一整个 renderer 的风格，而不是只处理单一页面
- 允许按 `shadcn` 常见后台应用模式重排页面布局
- 允许把现有页面拆分、组合或重命名为更清晰的视图层结构

### 3. 信息架构方向

- 一级导航固定为：
  - `作品`
  - `新建作品`
  - `设置`
- `新建作品` 和 `设置` 都作为常驻左侧导航中的一级视图
- `作品` 页面采用列表详情模式，而不是卡片墙或纵向长页面

## 当前问题

当前 renderer 已经引入 `shadcn` 配置和 Tailwind，但整体上仍处于“主题已接入、结构未统一”的状态。

主要问题：

- `renderer/components/ui/*` 中多个基础组件是手写的仿 `shadcn` 实现，不是官方生成源码
- `renderer/index.css` 中存在较强的自定义主题变量和面板阴影，视觉重心偏离 `shadcn default + slate`
- `App.tsx` 将 `Dashboard`、`NewBook`、`Settings`、`BookDetail` 以长页面方式连续堆叠
- `BookCard` 更接近概览卡片，不适合作为主列表项承担“选择当前作品”的职责
- `BookDetail` 的使用方式依赖页面底部追加 section，不符合桌面工作台的高频浏览节奏

结果是：组件来源不统一，布局语法不统一，页面之间缺少清晰的主次层级。

## 设计原则

### 1. 组件来源统一

所有基础交互元素优先使用 `shadcn/ui` 官方组件。若保留本地组件，也应明确为“业务组件”而不是“新的基础 UI 体系”。

### 2. 工作台优先

应用应该表现为稳定、连续、可长时间使用的桌面工具，而不是由多个展示块拼接而成的单页。

### 3. 列表与详情解耦

作品浏览和作品操作应该在同一工作区内完成，但承担不同职责：

- 列表负责发现、筛选、选择
- 详情负责阅读、控制、导出、删除

### 4. UI 重构不牵动业务逻辑

本次以前端结构和组件统一为主，尽量不改动 IPC 契约、存储层和写作引擎逻辑。

### 5. 避免二次视觉发明

本次不追求“再设计一套更酷的主题”，而是尽量靠近 `shadcn` 官方默认语言，只做必要的桌面化布局调整。

## 范围

### In Scope

- 统一 renderer 层主题与组件风格
- 安装并使用缺失的 `shadcn` 基础组件
- 重构应用外壳为左侧导航式布局
- 将 `作品` 页面重构为列表详情结构
- 将 `新建作品` 和 `设置` 重构为独立主视图
- 更新与新结构相关的 renderer 测试

### Out of Scope

- 不改动数据库 schema
- 不改动主进程写作逻辑
- 不修改 IPC channel 语义
- 不引入新的业务功能，如搜索、筛选、批量导出、模型分组等
- 不在本次引入暗色模式或多主题切换

## 整体壳子

应用整体布局改为三层：

1. `SidebarProvider` 作为外层状态容器
2. `AppSidebar` 承载一级导航
3. `SidebarInset` 承载当前主视图内容

左侧导航固定显示：

- `作品`
- `新建作品`
- `设置`

主内容区根据当前视图切换：

- `作品`：列表 + 详情
- `新建作品`：独立创建表单
- `设置`：独立配置视图

## 作品视图

作品视图采用 master-detail 结构。

### 左侧：作品列表面板

职责：

- 展示作品总览指标
- 承载批量操作
- 展示作品列表
- 负责选中当前作品

结构建议：

- 顶部标题和说明
- 统计摘要区
- 批量操作区：`全部开始`、`全部暂停`
- 作品列表滚动区

列表项信息至少包含：

- 标题
- 状态
- 章节完成信息
- 写作进度

列表项形态应更接近可选中的 row/card item，而不是独立大卡片墙。

### 右侧：作品详情面板

职责：

- 展示当前选中作品的完整上下文
- 承载单书操作
- 切换大纲、人物、章节、伏笔等内容区

结构建议：

- 详情头部：标题、状态、字数、主要操作
- 主体内容：`Tabs`
  - `大纲`
  - `人物`
  - `章节`
  - `伏笔`
- 内容区使用 `ScrollArea` 处理长内容

无选中作品时，右侧显示统一 empty state。

## 新建作品视图

`NewBook` 不再夹在 dashboard 旁边，而是独立成为一级主视图。

结构建议：

- 页面标题与简短说明
- 单张主 `Card`
- 表单内容保留当前字段：
  - `idea`
  - `targetWords`
- 表单节奏采用标准 `label + control + action`

成功创建后：

- 自动切回 `作品`
- 自动选中新创建的作品
- 右侧详情展示该作品最新状态

## 设置视图

`Settings` 也调整为独立主视图。

结构建议：

- 页面标题与说明
- 模型列表区域
- 模型编辑/测试区域
- 全局设置区域

视图组织上优先保证“模型列表”和“模型编辑”关系清晰，而不是继续通过一组按钮挤在同一信息密度层级内。

## 组件策略

### 直接采用 shadcn 官方组件

优先安装并使用：

- `sidebar`
- `button`
- `input`
- `textarea`
- `select`
- `label`
- `card`
- `separator`
- `scroll-area`
- `tabs`
- `badge`
- `alert`

根据实际实现情况可补充：

- `dialog`
- `sheet`
- `skeleton`

### 保留为业务组件的本地组件

这些组件仍可保留，但需要基于 `shadcn` 基础件重构表现层：

- `BookCard` 或其替代列表项组件
- `StatusBadge`
- `ProgressBar`
- `ChapterList`
- `ModelForm`
- `EmptyState`

判断标准：

- 若组件承载业务语义，应保留为本地业务组件
- 若组件只是在重复基础样式，应直接退回 `shadcn` 官方组件

## 主题策略

`renderer/index.css` 和相关 token 应收敛到接近 `shadcn default + slate` 的官方组织方式。

具体原则：

- 保留 CSS variables 方案
- 保留 `slate` 作为基础色来源
- 弱化当前过强的自定义阴影、超大圆角和高辨识度装饰性表面
- 尽量使用 `shadcn` 默认语义 token 命名和层级

允许保留的少量定制：

- 适配桌面应用的页面间距
- 适配双栏布局的容器尺寸
- 长文本内容区域的可读性 spacing

不再继续扩大自定义 token 集合。

## 交互规则

### 应用启动

- 默认进入 `作品`
- 若当前会话中已经存在 `selectedBookId`，则继续显示该书详情
- 若当前会话中尚未选中过书且列表非空，则默认选中第一本书
- 若没有书籍，右侧详情区显示 empty state，引导用户前往 `新建作品`

本次不要求把“当前选中作品”持久化到应用重启之后。

### 一级导航切换

- 切换 `作品 / 新建作品 / 设置` 时不销毁核心本地状态
- 返回 `作品` 时保留上次选中的书

### 作品列表交互

- 点击列表项只更新右侧详情，不跳页
- `全部开始 / 全部暂停` 保留在列表页头，而不是详情页

### 作品详情交互

单书操作集中在详情头部：

- `暂停`
- `恢复写作`
- `重新开始`
- `写下一章`
- `连续写作`
- `导出 TXT`
- `导出 MD`
- `删除作品`

这些操作继续复用现有 IPC 调用逻辑。

### 新建作品交互

- 创建提交后显示进行中反馈
- 成功后刷新书籍列表
- 切换回 `作品`
- 将新作品设为当前选中项

### 设置交互

- 模型列表支持选择当前模型
- 选择已保存模型后，右侧或相邻区域展示编辑表单
- 保存、测试、删除逻辑保持不变
- 并发设置保留独立保存逻辑

## 数据与状态边界

本次仅重组 renderer 视图层，不主动重构下列逻辑：

- `loadBooks`
- `loadBookDetail`
- `loadModels`
- `useIpc`
- `useProgress`
- 书籍调度、继续、暂停、重启、导出、删除 IPC 行为
- 模型保存、测试、删除与设置保存行为

允许的最小业务层变动仅限：

- 为了前端类型清晰做小范围类型整理
- 为了视图切换提取少量派生状态或本地 UI state

## 文件级改造方向

### `renderer/App.tsx`

- 重构为应用壳子与主视图切换中心
- 管理一级导航状态
- 管理当前选中作品
- 负责把现有动作分发到新视图

### `renderer/pages/Dashboard.tsx`

- 不再作为“整页 dashboard”存在
- 可拆分为作品列表头部和统计区域的局部视图，或由新组件取代

### `renderer/components/BookCard.tsx`

- 重构为适合主列表选择的列表项组件
- 强化选中态、状态信息和紧凑信息层级

### `renderer/pages/BookDetail.tsx`

- 保留业务内容组织
- 重构为右侧详情面板语义
- 标准化头部操作区、tabs 和内容滚动区

### `renderer/pages/NewBook.tsx`

- 保留现有表单字段
- 重构为独立创建视图

### `renderer/pages/Settings.tsx`

- 重构为独立配置视图
- 优化模型列表与编辑区域的布局关系

### `renderer/components/ui/*`

- 尽量替换为 `shadcn` 官方生成版本
- 删除或收缩不再需要的手写基础组件差异

## 测试策略

重点验证以下行为不回退：

- 左侧导航可在 `作品 / 新建作品 / 设置` 间切换
- 返回 `作品` 时保留当前选中书籍
- 点击作品列表项后，右侧详情更新
- 新建作品成功后切回 `作品` 并选中新作品
- 模型保存、测试、删除流程仍可操作
- 并发设置保存仍可操作
- 无书籍、无模型、无详情内容时 empty state 正常显示

优先更新 renderer 级测试，不将本次 UI 改造扩散到无关核心测试中。

## 实施顺序

1. 安装和补齐所需 `shadcn` 基础组件
2. 收敛全局主题 token 与基础 `ui` 组件实现
3. 重构 `App.tsx` 为 `Sidebar + Main View` 外壳
4. 实现 `作品` 视图的列表详情结构
5. 重排 `NewBook` 和 `Settings` 为独立主视图
6. 更新和补充测试
7. 执行 `typecheck` 与 `vitest`

## 风险与缓解

### 风险 1：UI 结构重排影响现有测试

缓解：

- 优先以行为断言替代过度依赖 DOM 结构的断言
- 将测试重心放在视图切换和关键操作路径

### 风险 2：手写 UI 组件与 shadcn 官方实现并存时间过长

缓解：

- 本次明确把“基础组件来源统一”纳入范围
- 不接受只改页面布局、不改基础组件来源的半迁移状态

### 风险 3：重排作品页时丢失部分操作入口

缓解：

- 将批量操作明确归属列表区
- 将单书操作明确归属详情区
- 实施时以操作清单逐项对照验收

## 验收标准

本次设计实施完成后，应满足：

- 用户一进入应用就能看到标准左侧导航工作台
- `作品` 页面是清晰的列表详情结构，而不是纵向堆叠页面
- `新建作品` 和 `设置` 是一级独立视图
- 基础组件风格一眼可辨识为 `shadcn default + slate`
- 不再明显存在“手写仿 shadcn”和“官方 shadcn”混用的视觉断层
- 原有书籍写作、详情查看、模型配置、调度和导出流程继续可用
