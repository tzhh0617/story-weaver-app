# Story Weaver 多书本双循环自动生成设计

## 背景

Story Weaver 当前已经具备单书自动生成的基础链路：从创意生成标题、世界观、总纲、章节，再逐章写作，并在写后抽取摘要、人物状态、伏笔和场景记录。

这条链路对短中篇已经够用，但要稳定支撑超长篇全自动生成，还存在三个核心问题：

- 规划粒度单一。远期目标、近期执行和章节产出混在一起，后期容易失控。
- 连续性依赖上下文拼接。模型一旦漂移，系统缺少明确的滚动纠偏点。
- 调度粒度以单书为主。未来如果要支持多书本自动生产，需要把“书”作为一等调度对象，而不是默认全局只有一本在跑。

本设计把系统升级为 `多书本 + 双循环 + 状态回写` 的自动生成架构。

双循环的含义是：

- `规划循环` 负责持续把远期方向收敛为近期可执行计划。
- `写作循环` 负责按当前计划稳定生成章节、审计章节、回写状态。

系统不追求一开始就做成复杂的平台，而是优先构建一条可长期运行、可按书隔离、可滚动重规划的最小自动生产线。

数据库层面采用开发期策略：`允许直接调整现有 schema 与 repository 结构，不要求兼容旧数据，也不为了保留历史表结构而牺牲设计清晰度。`

## 目标

- 支持多书本并行存在，每本书拥有独立的规划状态、写作状态和连续性状态。
- 采用双循环自动生成架构，把长期规划与逐章写作解耦。
- 用分层规划支持超长篇持续生成，避免把整本书一次性写死。
- 让全书规划持续受最初标题与灵感约束，确保长期生成不偏题、不丢卖点。
- 让规划区间按实际章节进度、完成度和剩余承诺动态调整，而不是固定写死到某个章数模板。
- 让规划本身提前设计波动、转折、兑现和代价，防止章节进入长期过平状态。
- 用结构化状态快照替代对超长上下文的依赖，降低模型漂移风险。
- 在尽量少的新概念下完成设计，避免引入过重的代理编排和平台化复杂度。

## 非目标

- 不在本轮设计里实现多代理协同平台。
- 不实现复杂的可视化流程编排器或全图谱工作台。
- 不在本轮设计里引入跨书共享世界观、共享角色宇宙等联动能力。
- 不追求一次性生成完美全书正文，系统仍然以分层规划和逐章迭代为核心。

## 设计原则

1. 一本书必须是独立状态机。
   多书支持的前提不是“多任务跑起来”，而是每本书都能独立规划、独立写作、独立复盘、独立纠偏。

2. 远期和近期必须分层。
   远处只定方向，中段只定任务，近处只定执行，不能把 2000 章都细化成一个静态计划。

3. 写作不能依赖模型记忆。
   章节连续性、人物变化、支线推进和世界事实，必须回写到结构化状态中，供后续检索和注入。

4. 规划必须滚动更新。
   计划不是开书时一次性生成完毕，而是在固定节点根据已发生事实重新展开未来区间。

5. 自动化优先于完美控制。
   第一版优先保证系统可持续自动产出，不引入大量人工审批节点，不构建复杂的策略分叉树。

6. 不为旧数据库结构让步。
   如果现有表结构不适合双循环和多书本规划，就直接改表、删表、合表、重命名；开发期不要求保留旧数据兼容。

7. 所有规划必须持续扣题。
   标题、核心灵感、题眼、初始卖点不能只在开书时使用一次，而要在阶段规划、中段规划、章节规划和章节审计时反复校验。

8. 章节规划必须动态伸缩。
   10 章、50 章、200 章只是默认观察粒度，不是硬边界。系统必须根据当前章节位置、已完成目标、未回收承诺和剩余可用章节数，动态压缩或拉伸后续计划。

9. 防平必须前置到规划层。
   不能等写完章节再发现“这章太平”。阶段、中段和章节计划在生成时就必须明确压力、变化、兑现、代价和转折，不允许连续产出只有推进没有波动的计划。

## 当前链路缺口

当前实现已经具备：

- `books`：书本元数据与基础上下文。
- `chapters`：章节正文与章节纲。
- `character_states`：章节后人物状态。
- `plot_threads`：伏笔开启和回收。
- `scene_records`：场景记录。
- `scheduler`：并发任务调度。

但对于“多书本双循环自动生成”，还缺少以下能力：

- 缺少按书隔离的长期规划对象，当前大纲和章节纲更偏线性写作输入。
- 缺少“远期粗、中期中、近期细”的分层计划结构。
- 缺少明确的固定复盘节点，例如每 10 章、50 章、200 章的自动再规划。
- 缺少把规划循环和写作循环拆开的状态机设计。
- 缺少多书视角下的任务公平调度策略，例如不同书本的章节生产和重规划如何排队。

此外，当前 `story_bibles`、`volume_plans`、`chapter_cards` 等结构虽然能承载部分规划信息，但它们是围绕上一阶段叙事控制闭环设计的，不一定天然适合新的双循环模型。本设计不要求在这些表之上“打补丁式扩展”，允许直接重构为更贴近双循环的数据模型。

同时，当前链路缺少两个核心控制点：

- 缺少对 `标题 / 灵感 / 核心卖点` 的持续校验机制，后续规划容易越写越偏。
- 缺少基于 `已写章节数 / 剩余章节预算 / 当前完成度` 的动态规划能力，导致固定章数切分在长篇后期容易失真。

另外，当前链路还缺少第三个控制点：

- 缺少在规划阶段显式设计 `压力上升 / 冲突切换 / 阶段兑现 / 不可逆变化` 的机制，容易在持续自动生成时出现章节长期过平。

## 总体架构

目标流水线如下：

```text
book created
  -> register title and idea contract
  -> generate endgame plan
  -> generate stage plans
  -> expand near-term arc plans
  -> expand first chapter plans
  -> write next chapter
  -> audit chapter draft
  -> revise or retry if needed
  -> extract and persist state snapshot
  -> trigger replanning on chapter milestones
  -> continue until target chapters complete
```

系统拆成两条循环：

### 规划循环

规划循环负责把长期目标逐步压缩成近期执行计划。

它只关心四层东西：

- `标题与灵感契约`
- `终局计划`
- `阶段路线`
- `中段计划`
- `章节计划`

规划循环不会直接产出正文，它的职责是给写作循环稳定供给下一段可写范围。

### 写作循环

写作循环负责逐章产出。

它只做四件事：

- 读取当前最小必要上下文
- 生成当前章节草稿
- 审计并修订草稿
- 回写章节状态快照

写作循环不会改远期规划；它只把“已经发生了什么”准确交回给规划循环。

## 分层规划模型

本设计使用三层渐进展开模型：

### 终局计划

终局计划是全书最高层约束，用来保证最终收束方向稳定。

它只保存少量长期不可轻易变化的信息：

- 主角终点
- 最终冲突
- 最终对手
- 世界最终状态
- 核心角色归宿
- 必须回收的重要承诺

终局计划不展开具体桥段，也不绑定具体章节。

但终局计划必须明确回应最初标题和灵感，回答：

- 这本书最初 promised 的是什么
- 最后兑现的还是不是这件事
- 终局是否仍然服务原始题眼和卖点

### 阶段路线

阶段路线对应远期粗规划，默认可按约 200 章一个阶段组织，但章节范围必须允许动态调整。

每个阶段保存：

- 阶段章节范围
- 阶段目标
- 阶段主要阻力
- 阶段升级点
- 阶段高潮
- 阶段回报
- 下一阶段悬念
- 本阶段如何持续兑现标题与灵感
- 本阶段最晚必须完成到什么进度，否则需要压缩后续章节预算
- 本阶段的主要压力曲线
- 本阶段至少一次不可逆变化
- 本阶段阶段性兑现点

阶段路线只提供方向，不提供逐章执行细节。

### 中段计划

中段计划对应中期规划，默认可按 50 章为一个区间组织，但区间长度要允许根据实际进度伸缩。

每个中段只回答未来这 50 章的任务：

- 主推进线有哪些
- 哪些角色要发生变化
- 哪些支线要推进或回收
- 本段结尾必须交付什么结果
- 本段如何继续扣住标题与灵感
- 如果实际推进过快或过慢，本段允许调整到多少章结束
- 本段压力如何升级
- 本段在哪里发生转折
- 本段必须兑现什么，不能只有铺垫
- 本段结束后必须留下什么新的不稳定状态

中段计划比阶段路线细，但仍然不细化到每章。

### 章节计划

章节计划是近区执行计划，是写作循环的直接输入。

建议仍按 10 章一个规划窗口组织，但对象命名统一成 `chapter plan`，不再保留 `batch` 作为核心命名。

每条章节计划保存：

- 所属规划窗口
- 本章目标
- 本章冲突
- 本章推进线
- 本章信息揭示
- 本章结尾钩子
- 本窗口必须兑现点
- 本窗口禁止偏离点
- 本章与标题/灵感的关联点
- 本章压力来源
- 本章变化类型
- 本章回报或代价

逐章正文只从章节计划中取任务，而不是直接从全书大纲展开。

## 核心对象

第一版只保留 5 个核心对象，避免设计膨胀。

这些对象是逻辑上的最小集合，不意味着数据库必须在保留旧表的前提下追加新表。实现时可以直接把现有规划类表改造成这 5 类核心对象的存储载体，或者删除不再合适的旧表后重建更清晰的表结构。

### 1. EndgamePlan

作用：保存一本书的终局约束。

建议字段：

```ts
type EndgamePlan = {
  bookId: string;
  titleIdeaContract: string;
  protagonistEndState: string;
  finalConflict: string;
  finalOpponent: string;
  worldEndState: string;
  coreCharacterOutcomes: Array<{
    characterId: string;
    outcome: string;
  }>;
  majorPayoffs: string[];
  updatedAt: string;
};
```

### 2. StagePlan

作用：保存一本书的远期阶段路线。

```ts
type StagePlan = {
  bookId: string;
  stageIndex: number;
  chapterStart: number;
  chapterEnd: number;
  chapterBudget: number;
  objective: string;
  primaryResistance: string;
  pressureCurve: string;
  escalation: string;
  climax: string;
  payoff: string;
  irreversibleChange: string;
  nextQuestion: string;
  titleIdeaFocus: string;
  compressionTrigger: string;
};
```

### 3. ArcPlan

作用：保存一本书的中段计划。

```ts
type ArcPlan = {
  bookId: string;
  arcIndex: number;
  chapterStart: number;
  chapterEnd: number;
  chapterBudget: number;
  primaryThreads: string[];
  characterTurns: string[];
  threadActions: string[];
  targetOutcome: string;
  escalationMode: string;
  turningPoint: string;
  requiredPayoff: string;
  resultingInstability: string;
  titleIdeaFocus: string;
  minChapterCount: number;
  maxChapterCount: number;
  status: 'planned' | 'active' | 'completed' | 'superseded';
};
```

### 4. ChapterPlan

作用：保存一本书的近区逐章执行计划。

```ts
type ChapterPlan = {
  bookId: string;
  batchIndex: number;
  chapterIndex: number;
  goal: string;
  conflict: string;
  pressureSource: string;
  changeType: string;
  threadActions: string[];
  reveal: string;
  payoffOrCost: string;
  endingHook: string;
  titleIdeaLink: string;
  batchGoal: string;
  requiredPayoffs: string[];
  forbiddenDrift: string[];
  status: 'planned' | 'active' | 'completed' | 'superseded';
};
```

### 5. StoryStateSnapshot

作用：保存一本书写完某章后的最新结构化状态。

```ts
type StoryStateSnapshot = {
  bookId: string;
  chapterIndex: number;
  summary: string;
  titleIdeaAlignment: string;
  flatnessRisk: string;
  characterChanges: string[];
  relationshipChanges: string[];
  worldFacts: string[];
  threadUpdates: string[];
  unresolvedPromises: string[];
  stageProgress: string;
  remainingChapterBudget: number;
  createdAt: string;
};
```

## 多书本隔离模型

多书支持的关键不是“可以开很多任务”，而是以下三层都按书隔离：

### 数据隔离

所有规划对象和状态对象都必须以 `bookId` 为第一过滤条件。

这意味着：

- 一本书的终局计划只作用于该书。
- 一本书的阶段路线不会污染其他书的规划。
- 一本书的状态快照不会被其他书误用为上下文。

### 执行隔离

每本书都维护独立的生成进度：

- 当前章号
- 当前阶段
- 当前中段
- 当前章节计划窗口
- 当前运行状态

任意一本书暂停、失败、重试，都不应破坏其他书的运行状态。

### 调度隔离

调度器需要把“书本任务”当作一等调度单位。

建议任务类型最少分为：

- `book:plan:init`
- `book:plan:rebuild-chapters`
- `book:plan:rebuild-arc`
- `book:write:chapter`

这样多书并发时，系统可以清楚知道某本书是在补规划，还是在产正文。

## 规划循环设计

规划循环的职责是：`持续把未来压缩为近期可写任务。`

### 初始化

一本书创建后，先执行：

1. 注册标题与灵感契约
2. 生成终局计划
3. 生成阶段路线
4. 展开当前阶段前两个中段计划
5. 生成第一组章节计划

这里刻意只展开近区，不一次性细化全书。

### 每 10 章复盘

当某本书完成一个近区规划窗口后：

1. 汇总最近 10 章状态快照
2. 检查本窗口目标是否完成
3. 检查是否发生重大偏移
4. 生成下一组章节计划
5. 检查最近章节是否仍然扣住标题与灵感
6. 根据剩余章节预算决定下一窗口是继续按 10 章展开，还是缩短/拉长到更合适的长度
7. 检查最近窗口是否已经出现连续过平，如果是则强制提高下一窗口的压力和兑现密度

如果偏移较小，只更新下一组章节计划。  
如果偏移较大，则要求先检查当前 50 章中段计划是否仍成立。

### 每 50 章重规划

当某本书完成一个中段区间后：

1. 检查中段目标是否完成
2. 检查支线承诺是否有积压
3. 检查角色变化是否仍与阶段路线一致
4. 生成下一组中段计划
5. 废弃已失效的章节计划
6. 根据已用章节数、剩余章节数、未兑现承诺，动态调整下一中段的长度
7. 如果本中段兑现不足或变化不足，下一中段必须增加转折和不可逆后果

### 每 200 章阶段校正

当某本书完成一个阶段后：

1. 校验阶段高潮与阶段回报是否落地
2. 检查终局计划是否仍成立
3. 检查哪些重要承诺需要延后或提前
4. 生成下一阶段的中段计划入口
5. 检查阶段是否仍然服务原始标题与灵感，如果偏题则强制重规划
6. 检查本阶段是否整体过平，如果是则在下一阶段提升结构性波动

规划循环不需要复杂推理链，只需要在这些固定节点滚动展开。

## 写作循环设计

写作循环的职责是：`按最近计划稳定生成当前章节。`

### 取上下文

每次写章时，只读取当前最小必要集合：

- 当前章卡
- 当前章节计划窗口
- 当前中段计划
- 标题与灵感契约
- 相关角色卡
- 相关支线状态
- 最近 3 到 5 章摘要
- 最新状态快照

这里强调最小必要上下文，是为了避免模型被远期信息淹没。

### 生成草稿

根据当前章卡和最小必要上下文生成章节草稿。

生成阶段不直接改规划对象；它只负责把本章写出来。

### 章节审计

生成完成后自动检查：

- 是否完成本章目标
- 是否与当前中段方向一致
- 是否与角色和世界事实冲突
- 是否保留下一章的连接点
- 是否仍然围绕标题和初始灵感推进，而不是脱离题眼自我扩散
- 是否至少兑现了计划中的变化、回报或代价，而不是只有信息搬运

### 自动修订

若审计失败，允许有限次数的自动修订或重写。  
若连续失败，则把该书标记为需要重新生成近区计划，而不是无限重复写章。

### 状态回写

章节通过后，提取并保存新的状态快照：

- 本章摘要
- 本章是否扣题
- 本章是否过平
- 人物变化
- 关系变化
- 新增世界事实
- 支线推进
- 未兑现承诺
- 阶段进度
- 剩余章节预算

回写成功后才推进到下一章。

## 调度模型

为了支持多书本但不把调度系统做重，第一版建议保留简单任务队列模型。

### 任务粒度

仅支持以下任务：

- 初始化规划任务
- 中段重规划任务
- 章节计划重建任务
- 单章写作任务

### 公平性

调度器不必一开始就实现复杂优先级树，但至少要避免某一本书长期占满全部执行槽位。

建议最小策略：

- 每本书同一时刻只允许一个运行中的写作任务
- 规划任务优先级高于普通写作任务
- 多书并发时采用轮转式出队，避免单书垄断

### 故障恢复

若一本书某次任务失败：

- 记录失败阶段与失败原因
- 允许按任务类型重试
- 不中断其他书任务

## 与现有架构的映射

本设计尽量贴近现有 Story Weaver 结构，不额外引入新服务框架，但允许直接调整数据库和 repository 分层。

可以自然映射到现有层：

- `src/core/book-service.ts`
  - 增加按书驱动规划循环和写作循环的入口
- `src/core/scheduler.ts`
  - 增加面向多书任务的调度类型
- `src/storage/*`
  - 增加终局计划、阶段路线、中段计划、章节计划、状态快照的存储
- `src/core/engine.ts`
  - 从“单线生成器”升级为“按书推进双循环”

现有的 `character_states`、`plot_threads`、`scene_records` 可以作为候选复用材料，但是否保留以新模型是否清晰为准。如果它们继续存在，就升级为状态快照的组成材料；如果它们妨碍双循环落地，就直接重构。

## MVP 范围

为了避免过度设计，第一版只做以下能力：

- 多书本独立规划与独立写作
- 双循环推进
- 5 个核心对象的持久化
- 10/50/200 章固定节点复盘
- 标题/灵感持续校验
- 基于剩余章节预算的动态规划
- 在规划层前置防平设计
- 单章生成、单章审计、状态回写
- 多书任务基础轮转调度

## 数据策略

本轮采用明确的开发期数据策略：

- 允许破坏性修改现有数据库 schema。
- 不要求保留旧表数据。
- 不要求提供历史数据迁移脚本。
- 优先选择最清晰的目标模型，而不是最小改动路径。

如果旧结构和目标结构冲突，默认以目标结构为准。

## 明确不做什么

第一版不做：

- 多代理流水线
- 可视化规划编排器
- 跨书共享宇宙
- 自动人工混编工作流
- 高级分析报表
- 复杂优先级策略引擎

这些能力未来可以加，但不属于当前最小闭环。

## 风险与取舍

### 风险 1：近区计划频繁失效

若模型生成的章节实际偏移较大，章节计划可能很快过期。

取舍：

- 接受近区计划会滚动重建
- 不试图一次规划过长区间

### 风险 2：多书并发导致调度不稳定

如果所有书同时进入重规划或重写阶段，简单队列可能抖动。

取舍：

- 第一版只做基础轮转和任务优先级
- 暂不做复杂资源控制模型

### 风险 3：状态回写质量影响后续全部生成

若状态抽取错误，会污染后续章节上下文。

取舍：

- 把状态回写视为强约束节点
- 优先保证抽取结构稳定，而不是追求一次提取所有细节

## 总结

本设计把 Story Weaver 的自动写作能力从“单书线性生成链路”升级为“多书本双循环自动生成系统”。

它的关键不在于增加更多模型角色，而在于建立一个足够轻但足够稳的结构：

- 一本书一个独立状态机
- 规划始终围绕标题与灵感，不允许长期偏题
- 长期规划和短期写作解耦
- 远期粗、中期中、近期细，而且区间可按实际章节动态伸缩
- 波动、兑现、代价和转折在规划阶段就被设计出来，而不是事后补救
- 每章写后必须回写状态
- 每 10/50/200 章固定滚动重规划

这套设计可以在不引入重平台复杂度的前提下，让系统逐步具备多书本、超长篇、持续自动生成的基础能力。
